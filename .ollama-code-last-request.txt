A lettura, le priorità sarebbero queste.

**1. In modalità patch, rifiuta davvero i path vietati.**
Adesso `extract_patch_paths()` li mette in `rejected`, ma poi `apply_unified_patch()` riceve comunque tutta la patch. È il bug più serio.
In `apply_pass()` farei così:

```python
rel_paths, rejected = extract_patch_paths(patch_text)
result.rejected = rejected

if rejected:
    print(c(f"\n❌ Patch rejected: invalid paths: {', '.join(rejected)}", RED))
    result.valid_response = False
    return result

if not rel_paths:
    result.valid_response = False
    return result
```

Meglio ancora: applicare ogni block separatamente e scartare solo quelli invalidi.

**2. Le cancellazioni di file oggi rischiano di fallire in validazione.**
Se una patch elimina `foo.py`, `compute_changed_files()` lo segnala come cambiato, poi `validate_written_files()` prova a fare `py_compile` su un file che non esiste più.
Dentro `validate_written_files()` aggiungi subito:

```python
if not path.exists():
    continue
```

Oppure traccia esplicitamente `created / modified / deleted`.

**3. Farei un `git apply --check` prima dell’apply reale.**
Così separi bene “patch non valida” da “patch applicata ma poi fallita la validazione”.

```python
def check_unified_patch(root: Path, patch_text: str) -> tuple[bool, str]:
    cp = subprocess.run(
        ["git", "-C", str(root), "apply", "--check", "--whitespace=nowarn", "--recount", "--unidiff-zero", "-"],
        input=patch_text,
        text=True,
        capture_output=True,
    )
    if cp.returncode == 0:
        return True, ""
    output = ((cp.stderr or "").strip() or (cp.stdout or "").strip() or "git apply --check failed.")
    return False, output
```

e poi in `apply_pass()` prima di `apply_unified_patch()`.

**4. `/last` non salva nulla.**
Il comando legge `LAST_RESPONSE_FILE`, ma nel codice che hai incollato non vedo nessun punto dove lo scrivi.
Ti conviene aggiungere:

```python
def save_last_response(root: Path, response: str) -> None:
    p = root / LAST_RESPONSE_FILE
    try:
        p.write_text(response.rstrip() + "\n", encoding="utf-8")
    except Exception as exc:
        print(c(f"Unable to save {p}: {exc}", YELLOW), file=sys.stderr)
```

e chiamarlo subito dopo `response = call_ollama(...)`.

**5. Non salverei la config automaticamente a ogni avvio.**
In `main()` fai merge di config + env + CLI e poi fai subito `save_config(root, config)`.
Effetto collaterale: un `--ollama-url` temporaneo diventa persistente anche se non volevi.
Io salverei solo quando cambi qualcosa con i comandi slash:

* `/model`
* `/fallback-model`
* `/plan`
* `/dry`
* `/commit`
* `/engine`
* `/readme`

**6. `make_commit_message()` usa endpoint/modello diversi dal resto della sessione.**
Qui fai:

```python
summary = call_ollama(summary_prompt, default_ollama_url(), None, plan_mode=False)
```

quindi ignori `ollama_url` e `model` correnti. È meglio passargli quelli attivi, altrimenti il commit summary può finire su un endpoint diverso.

**7. Stai facendo molte chiamate extra a Ollama.**
Per ogni richiesta puoi avere:

* riassunto richiesta
* piano
* draft
* review
* fix
* polish
* commit summary

Funziona, ma rallenta molto.
Io renderei opzionali almeno due cose:

* `summarize_request_for_history()`
* `make_commit_message()` con fallback locale semplice

Per esempio, se vuoi velocità, il summary storico può essere solo `one_line_summary(request)` senza chiamare il modello.

**8. Scegli bene cosa è “per utente” e cosa è “per progetto”.**
Hai fatto bene a spostare config e history readline in `~/.ollama-code/`.
Però questi restano ancora nella root del progetto:

* `.ollama-code-last-request.txt`
* `.ollama-code-requests-history.txt`
* `.ollama-code-last-response.txt`

Non è sbagliato, ma conviene decidere chiaramente:

* **utente**: config, modello, endpoint, history del prompt
* **progetto**: last request, request history, last response

oppure spostare anche questi sotto una cartella dedicata del progetto, per esempio `root/.ollama-code/`.

**9. In patch mode, farei tracking dei cambiamenti usando Git, non solo il backup in memoria.**
`compute_changed_files()` confronta solo i path in `backup`. È semplice, ma con patch più complesse è meno robusto.
Alternativa migliore: dopo l’apply, usare `git status --porcelain -- <paths...>` per sapere davvero cosa è cambiato.

**10. Renderei i toggle meno ripetitivi.**
I rami `/plan`, `/dry`, `/commit`, `/engine`, `/readme` sono quasi identici.
Puoi centralizzare con una funzione tipo:

```python
def toggle_setting(config, root, key, current_value, arg, label):
    if arg.lower() == "on":
        current_value = True
    elif arg.lower() == "off":
        current_value = False
    else:
        current_value = not current_value
    config[key] = current_value
    save_config(root, config)
    print(c(f"{label}: {'ON' if current_value else 'OFF'}", GREEN))
    return current_value
```

**11. Aggiungerei anche i flag inversi simmetrici da CLI.**
Adesso hai:

* `--plan`
* `--dry-run`
* `--no-commit`
* `--no-engine`
* `--no-readme`

Mancano `--no-plan` e `--no-dry-run`.
Con `argparse.BooleanOptionalAction` diventa più pulito se la tua versione Python lo supporta.

**12. Per la patch mode, il salto successivo è applicare block-by-block.**
È il miglioramento architetturale più utile. Ti permette di:

* rifiutare un solo patch block
* fare backup preciso per file
* avere errori più leggibili
* capire meglio cosa ha provato a modificare il modello

Le tre cose che farei subito, senza dubbio, sono:

1. bloccare davvero i path invalidi prima di `git apply`
2. saltare la validazione dei file cancellati
3. salvare davvero `LAST_RESPONSE_FILE`

Queste tre aumentano molto l’affidabilità senza cambiare troppo la struttura.
