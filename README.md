# oc

`oc` is a local terminal coding agent for Ollama. It is the direct successor to `ollama-code`, but it keeps the short command name `oc`.

The project is intentionally simple: the current implementation is a single Python workflow that reads a request from the workspace, inspects the repository, asks an Ollama model for a plan and file actions, applies those changes, and runs a small set of automatic checks.

## How It Works

Every run follows the same flow:

1. `oc` loads or creates a local `.config` file with the Ollama endpoint, model, and temperature.
2. It opens `specs.txt` in your editor so you can write or update the task.
3. It scans the current workspace and collects a bounded text context for the model.
4. If code already exists, it runs an initial validation pass and asks the model for a short analysis of the current state.
5. It asks the model for a 3-6 step plan.
6. It asks the model for structured JSON actions:
   - `write_file`
   - `append_file`
   - `delete_file`
7. It applies the returned file operations.
8. It runs automatic checks on the touched files.
9. It writes `report.txt` with the last iteration summary.
10. If checks fail, it retries the planning and generation loop up to 3 iterations.

The model is only allowed to work inside the current repository. Protected paths include `.git`, `.config`, `specs.txt`, `report.txt`, and `README.md`.

## Current Behavior

- The tool always works from the current directory.
- The user request is sourced from `specs.txt`, not from command-line free text.
- `README.md` is generated by the tool in the active workspace on each run.
- If the workspace contains no code yet, `oc` switches to bootstrap mode and generates a minimal initial structure.
- The terminal UI, prompts, report text, and model instructions are currently in Italian.

## Validation

`oc` performs only lightweight validation, based on the files changed in the last iteration:

- Python: `python -m py_compile`
- JavaScript: `node --check` for `.js`, `.mjs`, `.cjs` when Node.js is available
- JSON: standard JSON parsing
- HTML: `tidy -qe` when available, otherwise a basic built-in HTML5 check
- Shell: `bash -n` for `.sh` files when Bash is available
- Tests: `pytest -q` only when Python files changed, `pytest` is installed, and the project looks like a pytest project

## Files Used By The Workflow

- `.config`: local Ollama configuration in JSON format
- `specs.txt`: the task you want `oc` to execute
- `report.txt`: the last execution report
- `README.md`: generated workspace overview
- `oc`: Unix-like launcher
- `oc.py`: Python entrypoint
- `oc.bat`: Windows launcher

Example `.config`:

```json
{
  "base_url": "http://127.0.0.1:11434",
  "model": "llama3.1:8b",
  "temperature": 0.2
}
```

## Requirements

- Python 3.9+
- [Ollama](https://ollama.com/) installed and running
- At least one local Ollama model available

Optional tools used when present:

- `node` for JavaScript syntax checks
- `pytest` for Python test runs
- `bash` for shell syntax checks
- `tidy` for stricter HTML validation

## Installation

Clone the repository:

```bash
git clone https://github.com/your-name/oc.git
cd oc
```

### Linux and macOS

Use the executable script directly:

```bash
chmod +x oc
./oc
```

To make it globally available:

```bash
install -m 755 oc ~/.local/bin/oc
```

### Windows

Keep `oc.py` and `oc.bat` in the same directory, then run:

```bat
oc.bat
```

Or directly with Python:

```bat
py -3 oc.py
```

To make `oc` available everywhere on Windows, place that directory in your `PATH`.

## Usage

Run `oc` inside the repository you want to modify:

```bash
oc
```

On the first run, `oc` creates `.config` interactively. After that it:

1. opens `specs.txt`,
2. reads your request,
3. talks to Ollama,
4. applies the generated changes,
5. validates the result,
6. writes `report.txt`.

To reset the local configuration:

```bash
oc -r
```

## Windows Compatibility

The current `oc.py` + `oc.bat` setup is compatible with Windows with these notes:

- `oc.bat` launches the adjacent `oc.py` file and forwards CLI arguments correctly.
- The Python script can use `VISUAL` / `EDITOR` on Windows and includes `notepad` as a fallback editor.
- Shell validation for `.sh` files is skipped unless Bash is installed.
- HTML validation falls back to the built-in parser if `tidy` is not installed.

This was verified statically from the current source code. No native Windows runtime was available in this environment for an end-to-end execution test.

## Limitations

- The project is not packaged as an installable Python module yet.
- The workflow is intentionally narrow: one request file, one workspace, one model configuration.
- Large repositories are truncated before being sent to the model.
- Output quality depends heavily on the selected Ollama model.
- `README.md` is workspace-generated, so running `oc` in this repository will rewrite the repository README.
